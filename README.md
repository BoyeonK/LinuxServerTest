# windows로는 AWS EC2 프리티어 쌀먹을 할 수가 없어!

- 메인프로세스
  - 매치메이킹 및 각 프로세스간 IPC를 담당
  - 활동 유저 수가 증가한 경우 유동적으로 Dedicate Process를 늘리고 줄임.

- 게임 서버(Dedicate)
  - 별도의 프로세스에서 실행됨.
    - 메인 프로세스와 Unix Domain Socket으로 IPC구성
  - 클라이언트와 UDP통신.
  - 시작된 인게임에서의 로직과 통신을 담당.

- HTTPS서버
  - 별도의 프로세스로서 실행됨.
    - 메인 프로세스와 Unix Domain Socket으로 IPC구성
  - node.js로 작성됨.
    - https통신을 사용함. 클라이언트와 접촉은 Cloudflare를 통함.
    - 게임 서버에서 MySQL을 조작해야 할 경우, HTTPS서버를 통함.
    - 게임을 매칭하기 이전, 로비에서의 요청 담당 (속도보다 정합성이 중요한 경우)
  - 스키마는 http-api-spec.yaml에 문서화 되있으며, Swagger로 시각화
  
- DB
  - MySQL
    - 정합성이 중요한 자료의 경우. (유저정보, 인벤토리)
    - HTTPServer/database/schema.sql에 사용할 테이블 문서화
  - Redis
    - 주로 C++쪽에서 DB에 접근할 경우 캐싱 (메인프로세스 혹은 Dedicate서버에서 접근)
    - 날아가도 상관없는(?)거 (예 : 게스트 유저 정보)
    - HTTP서버에서 조작할 일도 있으며, race condition이 발생할 가능성이 있는 모든 경우에서 Lua사용
    - 일부 자료의 경우, 프로세스간(Node.js <-> C++) 공유 메모리 역할로 사용중. (예 : 유저의 매치메이킹 진행 현황 정보)
      - 이 경우, 최대한 source of truth를 이 데이터로 사용.
    - HTTPServer/database/redis_keys.md에 사용할 스키마 문서화.
